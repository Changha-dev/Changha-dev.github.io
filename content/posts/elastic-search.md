# 들어가며

나르지지 홈페이지에 검색 기능을 추가하면서, 사용자가 팀 이름만 입력해도 관련 경기들을 빠르게 찾을 수 있는경험을 만들고 싶었습니다.

특히 `T1 vs GEN`처럼 두 팀의 대진을 바로 검색해 최신 경기 기록을 보여주면 편리하겠다고 생각했고, 이를 검색 자동완성 형태로 구현하기로 했습니다.

![image](../elastic-search-img/img1.png)

## 요구사항

`팀 vs 팀` 검색은 단순 문자열 검색보다 조금 더 복잡했습니다.

사용자는 다음과 같이 다양한 형태로 검색할 수 있습니다.

- 영어 vs 영어: `T1 vs GEN`
- 한글 vs 한글: `티원 vs 젠지`
- 초성 vs 초성: `ㅌㅇ vs ㅈㅈ`
- 공백이나 구분자 없는 형태: `T1vsGEN`

즉, 검색어가 어떤 형태로 들어오더라도 결국 유효한 팀을 식별해내고,

그 팀 ID를 기반으로 MySQL에서 한 팀의 최신 경기 목록 또는 두 팀 간의 최신 맞대결 기록을 반환해야 했습니다.

검색어는 그대로 Elasticsearch에 전달한 것이 아니라, 먼저 애플리케이션 레벨에서 `vs`, `vs.`, `대`, 공백, `-` 등을 기준으로 분리해 팀 키워드를 추출한 뒤 각 키워드에 대해 팀 검색을 수행했습니다.

# 기술 스택 선택 과정

처음에는 별도 인프라 없이 기존 MySQL만 활용하는 방법을 우선 고려했습니다.

운영 비용과 관리 포인트를 늘리지 않는다는 점에서 분명 장점이 있었기 때문입니다.

다만 이번 검색은 단순히 팀 이름을 부분 일치로 찾는 수준이 아니라, 영문, 한글, 초성, 팀 코드, 자동완성을 함께 고려해야 했습니다.

MySQL도 FULLTEXT 같은 선택지가 있지만, 이번 요구사항에서는 검색어 정규화, 가중치 조절, 자동완성 대응, 한글 검색 품질 측면에서 Elasticsearch가 더 적합하다고 판단했습니다.

## MySQL 의 FULLTEXT

처음에는 별도 검색 엔진을 두지 않고 MySQL의 FULLTEXT 검색을 활용하는 방법을 먼저 검토했습니다.

FULLTEXT 검색은 일반적인 `LIKE '%keyword%'` 방식보다 검색용 인덱스를 활용할 수 있어, 텍스트 검색 성능과 검색어 매칭 측면에서 더 유리한 방식입니다. 특히 자연어 검색이나 boolean mode를 활용하면 단순 포함 검색보다 더 다양한 검색 조건을 줄 수 있습니다.

즉, 팀 이름 검색 정도라면 MySQL만으로도 어느 정도 구현은 가능하다고 봤습니다.

다만 이번 검색 기능은 단순히 팀 이름 하나를 찾는 수준이 아니었습니다.

- 영문 팀명 검색
- 한글 팀명 검색
- 초성 검색
- 팀 코드 검색
- `팀 vs 팀` 형태의 자동완성 검색

이처럼 서로 다른 형태의 입력을 하나의 검색 경험으로 묶어야 했고, 검색 결과의 우선순위도 세밀하게 제어할 필요가 있었습니다.

이 지점에서 MySQL FULLTEXT만으로는 구현 복잡도와 확장성 측면에서 아쉬움이 있다고 판단했습니다.

## 그래서 Elasticsearch는?

이번 검색 기능은 단순 문자열 검색이 아니라, 다양한 형태의 입력을 하나의 검색 경험으로 연결하는 작업에 가까웠습니다.

예를 들어 사용자는 다음과 같이 검색할 수 있습니다.

- `T1 vs GEN`
- `티원 vs 젠지`
- `ㅌㅇ vs ㅈㅈ`
- `T1vsGEN`

즉, 영문 팀명, 한글 팀명, 초성, 팀 코드처럼 서로 다른 표현을 모두 하나의 팀으로 식별할 수 있어야 했습니다.

또 검색 결과는 단순히 팀 목록을 보여주는 것이 아니라, 식별된 팀 ID를 기반으로 실제 경기 데이터와 연결되어야 했습니다.

이런 요구사항을 만족하려면 검색어를 단순 부분 일치로 처리하는 것보다, 검색 전용 인덱스를 설계하고 검색 결과의 우선순위를 제어할 수 있는 방식이 더 적합하다고 판단했습니다.

그래서 Elasticsearch를 도입했습니다.

# 아키텍처

![image](../elastic-search-img/img2.png)

Elasticsearch가 먼저 팀 후보를 식별하고, 이후 실제 경기 목록과 맞대결 기록은 MySQL에서 조회하는 방식으로 역할을 분리했습니다.

검색 엔진은 `무엇을 찾을지`를 결정하고, 관계형 데이터베이스는 `찾은 팀으로 어떤 경기들을 보여줄지`를 담당하도록 설계한 셈입니다.

즉, 이 검색 기능은 Elasticsearch만으로 완결되는 구조가 아니라,

Elasticsearch가 먼저 팀 후보를 식별하고, 이후 실제 경기 데이터는 MySQL에서 조회하는 구조입니다.

# Elasticsearch에서 어떻게 설계했는가

Elasticsearch에는 팀과 선수 정보를 검색용 문서로 별도 저장했습니다.

팀 문서에는 다음과 같은 필드를 포함했습니다.

- 영문 팀명
- 한글 팀명
- 정규화한 이름
- 자동완성용 필드
- 팀 코드

예를 들어 `Gen.G`라는 팀은 원본 이름 그대로 저장하는 것뿐 아니라, 특수문자를 제거한 `geng` 형태로도 검색할 수 있도록 정규화 필드를 따로 두었습니다.

또 한글 검색 품질을 높이기 위해 한글 팀명을 별도로 매핑했고, 초성 검색을 위해 `티원 -> ㅌㅇ`, `젠지 -> ㅈㅈ` 같은 값을 자동완성 필드에 함께 넣었습니다.

여기서 자동완성 필드는 Elasticsearch가 자동으로 만들어주는 값이 아니라, 검색에 필요한 표현을 직접 조합해 만든 필드였습니다.

영문 팀명, 한글 팀명, 초성, 팀 코드를 하나의 문자열로 구성해 저장했고, 여기에 `edge ngram` analyzer를 적용해 사용자가 검색어를 끝까지 입력하지 않아도 후보를 찾을 수 있도록 설계했습니다.

이렇게 구성해두면 사용자가 어떤 표현으로 검색하더라도 Elasticsearch가 먼저 가장 적절한 팀 후보를 찾고, 그 결과를 다시 MySQL의 경기 데이터 조회로 연결할 수 있습니다.

## 검색 품질을 위해 적용한 설정

한글 검색을 위해 Elasticsearch의 `Nori analyzer`를 사용했고, 자동완성을 위해 `edge ngram` 기반 필드도 함께 구성했습니다.

`Nori analyzer`는 한글 형태를 분석하는 데 유리하고, `edge ngram`은 검색어를 앞부분 기준으로 잘라 색인하기 때문에 자동완성 구현에 적합했습니다.

또 검색 정확도를 높이기 위해 모든 필드를 동일하게 취급하지 않고 우선순위를 두었습니다.

예를 들어 팀 검색 쿼리에서는 다음과 같이 가중치를 다르게 설정했습니다.

- `team_code` 정확 일치: `boost 5.0`
- 영문 이름 match: `boost 3.0`
- 한글 이름 match: `boost 2.0`
- 정규화 이름 prefix: `boost 2.0`
- 자동완성 필드 match: `boost 1.0`

이렇게 해야 `T1`, `GEN`처럼 짧고 명확한 입력에서 사용자가 의도한 팀이 상단에 노출될 가능성을 높일 수 있었습니다.

# 시행착오

## 검색 범위를 어느 정도로 설정할 것인가

처음에는 단순하게 접근했습니다. "데이터가 많으면 많을수록 사용자가 원하는 팀을 찾을 확률이 높지 않을까?"라는 생각에, DB에 등록된 모든 팀(약 538개)을 인덱싱하고 검색 후보로 두었습니다. 하지만 실제 테스트를 해보니 기대와는 다른 결과가 나타났습니다.

### **1. edge_ngram의 양날의 검**

자동완성 기능을 위해 도입한 `edge_ngram`이 문제였습니다. 이 방식은 검색어의 앞부분만 입력해도 결과를 미리 보여주기 때문에 사용자 편의성은 높지만, 검색 범위가 너무 넓으면 오히려 독이 되었습니다. 예를 들어, 사용자가 'T1'을 검색했을 때 'T1'과 관련 없는 수많은 유사 이름의 하위 리그 팀들이 함께 매칭되면서, 정작 사용자가 보고 싶어 하는 '주요 팀'이 순위에서 밀려나는 현상이 발생한 것이죠.

### **2. 데이터 품질의 격차: 538개 대 68개**

결정적인 원인은 데이터의 불균형에 있었습니다. 현재 DB 기준으로 전체 팀은 538개였지만, 유명한 리그에 포함되는 팀 식별 가능한 `team_code`를 가진 팀은 68개뿐이었습니다.
나머지 팀들은 비주류 리그라서 검색에서 바로 식별할 수 있는 명확한 메타데이터를 포함하지 않았습니다. 이런 팀들이 검색 결과에 섞이면서 전체적인 검색 정확도를 떨어뜨리고 있었습니다. 사용자가 '젠지(GEN)'를 검색할 때, 이름도 생소한 하위 리그의 유사한 팀이 먼저 튀어나오는 상황은 결코 좋은 경험이 아니었습니다.

### **3. 검색 품질을 높이기 위한 두 가지 전략**

결국 "무조건 많이" 보여주는 것보다 "정확하게" 보여주는 것이 우선이라는 결론을 내렸습니다. 이를 위해 다음과 같은 보완책을 적용했습니다.

- **가중치 설계:** Elasticsearch 쿼리에서 필드별로 중요도를 다르게 설정했습니다. 가장 명확한 식별자인 `team_code` 일치에 최고 가중치를 주고, 그다음으로 영문명, 한글명, 정규화 이름 순으로 점수를 차등 부여했습니다. 이렇게 하니 'T1'이나 'GEN' 같은 명확한 키워드에서 엉뚱한 팀이 튀어나오는 일이 현저히 줄어들었습니다.
- **Top-1 매칭 전략:** Elasticsearch가 반환하는 여러 결과 중 점수가 가장 높은 **상위 1개 팀**만을 실제 경기 데이터 조회에 사용하도록 처리했습니다. 검색 엔진이 판단한 가장 '확실한' 후보 하나에 집중하도록 하였습니다.

# 느낀점

이번 작업을 통해 검색 서비스는 단순히 데이터를 찾아주는 것을 넘어, **사용자의 의도와 데이터의 품질 사이에서 균형을 잡는 최적화 과정**이라는 점을 다시 한번 깨달았습니다.

범위를 무작정 넓히기보다, 식별 가능한 메타데이터가 확실한 주요 팀들을 중심으로 검색 후보를 정교하게 다듬는 것이 실제 서비스의 사용자 경험(UX) 측면에서는 훨씬 더 유리하다는 값진 경험을 얻었습니다.
